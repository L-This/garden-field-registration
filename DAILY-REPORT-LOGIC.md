# منطق التقرير اليومي

## 1) نفّذ SQL أولًا

شغّل الملف التالي كاملًا داخل Supabase SQL Editor:

`supabase-daily-report-logic.sql`

ينشئ الملف:

- جدول `daily_report_submissions` لقفل تقرير المشروع لكل تاريخ.
- جدول `daily_report_windows` لفتح تقرير تعويضي بتاريخ سابق.
- ربط سجلات `reports` بالتقرير اليومي.
- حماية تمنع تقريرًا ثانيًا لنفس المشروع والتاريخ حتى من جهاز آخر.
- اكتشاف التقارير القديمة الموجودة مسبقًا وإغلاق الصفحة بناءً عليها.

## 2) السلوك بعد التحديث

- عند فتح صفحة المقاول، يتم سؤال Supabase عن حالة تقرير المشروع والتاريخ.
- إذا كان التقرير مرسلًا، تظهر رسالة «تم إنجاز المهمة» ولا يظهر نموذج الرفع.
- إذا كان الإرسال جاريًا من جهاز آخر، تظهر رسالة «التقرير قيد الإرسال».
- التقرير يُعتمد كعملية واحدة كاملة؛ إذا فشل موقع واحد، يُلغى التقرير الجزئي ولا يُعتمد.
- بعد نجاح التقرير، تُغلق صفحة الرفع لجميع الأجهزة.
- التطبيق يعيد التحقق كل دقيقة وعند الرجوع إلى التبويب، لذلك ينتقل تلقائيًا لجدول اليوم الجديد.

## 3) فتح يوم سابق للتعويض

احصل على UUID المشروع ثم نفّذ مثالًا مشابهًا:

```sql
insert into public.daily_report_windows (
  project_id,
  report_date,
  closes_at,
  note
)
values (
  'PROJECT_UUID',
  current_date - 1,
  now() + interval '6 hours',
  'فتح تعويض تقرير يوم سابق'
);
```

أثناء الفترة المفتوحة، صفحة المقاول تعرض تاريخ التعويض بوضوح، وبعد الإرسال تُغلق النافذة تلقائيًا.

## 4) إغلاق نافذة تعويض يدويًا

```sql
update public.daily_report_windows
set active = false
where project_id = 'PROJECT_UUID'
  and active = true;
```
